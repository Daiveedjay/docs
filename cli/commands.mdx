The Iris CLI provides eleven commands covering the full code health workflow â€” scanning individual files, auditing dependencies, exporting a software bill of materials, enforcing thresholds in CI, and managing hooks. Commands marked **Pro** require an active Iris Pro licence.

### iris check

Scores source files and prints health findings. Single-file scans are free; scanning a directory or using `--staged` / `--changed` requires Pro.

| Flag | Description |
|---|---|
| `[path]` | File or directory to scan (defaults to current directory) |
| `--staged` | Scan staged git files only **(Pro)** |
| `--changed` | Scan files changed since the last commit **(Pro)** |
| `--min-score <n>` | Minimum health score 0â€“100; overrides `.irisconfig.json` |
| `--config <path>` | Use a specific `.irisconfig.json` |
| `--format <fmt>` | Output format: `pretty` (default), `json`, or `github` |
| `--output <path>` | Write JSON result to file (requires `--format json`) |
| `--verbose` | Show individual findings per file |

```bash
# Score a single file (free)
iris check src/index.ts

# Score a directory with a custom threshold (Pro)
iris check src/ --min-score 80

# Score staged files only (Pro)
iris check --staged
```

| Exit code | Meaning |
|---|---|
| `0` | All files at or above threshold |
| `1` | One or more files below threshold |
| `2` | Bad arguments or config error |

### iris security

Scans the project for nine security anti-patterns: eval/exec usage, SQL built by string concatenation, insecure RNG, ReDoS-prone regex, hardcoded localhost URLs, disabled TLS verification, debug flags, weak hashing (MD5/SHA-1), and open redirects. See [Security Smells](/enforcement/security-smells) for the full pattern reference. **Free â€” no authentication required.**

| Flag | Description |
|---|---|
| `[path]` | File or directory to scan (defaults to current directory) |
| `--format <fmt>` | Output format: `pretty` (default) or `json` |
| `--output <path>` | Write JSON result to file (requires `--format json`) |

```bash
# Scan the current directory
iris security

# Scan a specific path and output JSON
iris security ./src --format json
```

| Exit code | Meaning |
|---|---|
| `0` | No security smells found |
| `1` | Security smells found |
| `2` | Bad arguments |

### iris deps

**Pro.** Audits dependencies in `package.json`, `go.mod`, `requirements.txt`, or `pyproject.toml` for outdated versions and known CVEs via the GitHub Advisory Database.

| Flag | Description |
|---|---|
| `[path]` | Directory to scan (defaults to current directory) |
| `--no-cache` | Skip the 24-hour local cache and re-fetch |
| `--verbose` | Show individual CVE titles and URLs |
| `--format <fmt>` | Output format: `pretty` (default) or `json` |
| `--output <path>` | Write JSON result to file (requires `--format json`) |

```bash
# Audit dependencies in the current project
iris deps

# Bypass the cache and output JSON
iris deps --no-cache --format json
```

Without a GitHub token, CVE lookups are limited to 60 req/hr. Run `iris auth login` to store a GitHub PAT, or set `GITHUB_TOKEN`, to raise it to 5,000 req/hr.

| Exit code | Meaning |
|---|---|
| `0` | No vulnerabilities found |
| `1` | One or more vulnerabilities detected |
| `2` | Invalid arguments or no manifest found |

### iris sbom

**Pro.** Exports a [CycloneDX](https://cyclonedx.org/) 1.5 Software Bill of Materials (SBOM) for the workspace â€” a standard, machine-readable dependency manifest increasingly required for supply-chain compliance and procurement. Generation is fully offline: it reads `package.json`, `go.mod`, `requirements.txt`, `pyproject.toml`, and lockfiles, and never makes a network call itself.

| Flag | Description |
|---|---|
| `[path]` | Directory to scan (defaults to current directory) |
| `--output <file>` | Write to a specific path (default: `iris-sbom.json` in the workspace) |
| `--stdout` | Print the SBOM to stdout instead of writing a file |

```bash
# Write iris-sbom.json in the current directory
iris sbom

# Write to a specific path
iris sbom --output sbom.json

# Print to stdout
iris sbom --stdout
```

Each component in the output includes its name, version, [package URL](https://github.com/package-url/purl-spec) (`purl`), and a `required`/`optional` scope marking direct vs. dev dependencies. If a cached [`iris deps`](#iris-deps) scan exists, its known-vulnerability findings are folded in as CycloneDX `vulnerabilities` entries, cross-referenced to the affected component. Run `iris deps` first if you want vulnerability data included â€” without it, the SBOM still generates, just without vulnerabilities.

| Exit code | Meaning |
|---|---|
| `0` | SBOM written successfully |
| `2` | Invalid arguments or no manifest found |

**Pro.** Lists every `TODO`, `FIXME`, and `HACK` comment found across the project. This command is informational â€” it does not fail the run based on findings.

| Flag | Description |
|---|---|

| `--format <fmt>` | Output format: `pretty` (default) or `json` |
| `--output <path>` | Write JSON result to file (requires `--format json`) |

```bash
# Scan a specific path and output JSON

```

| Exit code | Meaning |
|---|---|
| `0` | Success (regardless of findings) |
| `2` | Bad arguments |

### iris gate

**Pro.** Runs the full enforcement gate â€” every rule configured in `.irisconfig.json` (`minHealthScore`, `gateMaxSecrets`, `gateMaxComplexity`, `gateMaxFileLength`, `gateMaxSmellsPerFile`, `gateMaxSecuritySmells`) â€” and shows each rule's threshold vs actual value with per-rule PASS/FAIL. This is the recommended command for CI quality gates. There is no CLI flag to override thresholds; set them in `.irisconfig.json`.

| Flag | Description |
|---|---|
| `[path]` | File or directory to scan (defaults to current directory) |
| `--config <path>` | Use a specific `.irisconfig.json` |
| `--format <fmt>` | Output format: `pretty` (default), `json`, or `github` |
| `--output <path>` | Write JSON result to file (requires `--format json`) |

```bash
# Run the gate against the configured rules
iris gate

# Emit inline PR annotations in GitHub Actions format
iris gate . --format github
```

| Exit code | Meaning |
|---|---|
| `0` | All rules pass |
| `1` | One or more rules fail |
| `2` | Bad arguments or config error |

### iris hook

Installs and removes the git pre-push hook and the build gate hook â€” equivalent to the VS Code command palette hook commands. The type (`git` or `build`) comes before the action. `status` is **free**; `install` and `uninstall` require **Pro**.

| Subcommand | Description |
|---|---|
| `iris hook git status [path]` | Show whether the git pre-push hook is installed **(free)** |
| `iris hook git install [path]` | Write the Iris block to `.git/hooks/pre-push` **(Pro)** |
| `iris hook git uninstall [path]` | Remove the Iris block from `.git/hooks/pre-push` **(Pro)** |
| `iris hook build status [path]` | Show whether the build hook is installed **(free)** |
| `iris hook build install [path]` | Wire Iris into the project build command **(Pro)** |
| `iris hook build uninstall [path]` | Remove the Iris build entry **(Pro)** |

```bash
# Check current hook status (free)
iris hook git status

# Install the git pre-push hook
iris hook git install

# Remove the build hook
iris hook build uninstall
```

| Exit code | Meaning |
|---|---|
| `0` | Success |
| `2` | Bad arguments, not a git repository, or no project manifest found |

### iris auth

Manages authentication credentials. Every subcommand is fully interactive â€” there is no non-interactive flag to pass a token directly on the command line; use the `IRIS_LICENCE_TOKEN` environment variable instead for CI runners. See the [Authentication](/cli/auth) page for the full guide.

| Subcommand | Description |
|---|---|
| `login` | Interactive prompt: browser-based sign-in or paste a licence token. Requires a TTY. |
| `status` | Show the current authentication and GitHub token state |
| `logout` | Remove stored credentials |

```bash
# Sign in interactively
iris auth login

# Check the current session
iris auth status
```

### iris config

Generates or validates `.irisconfig.json`.

| Subcommand | Flag | Description |
|---|---|---|
| `init [path]` | â€” | Interactively pick a preset (`legacy`, `balanced`, `strict`) and write `.irisconfig.json`. Pro users are also prompted for gate limits. Falls back to a non-interactive `balanced` default when stdin is not a TTY (e.g. in CI). |
| `validate [path]` | â€” | Validate an existing `.irisconfig.json` and print any schema or value problems |

```bash
# Interactively generate a config
iris config init

# Validate an existing config
iris config validate
```

| Exit code | Meaning |
|---|---|
| `0` | Success / no validation issues |
| `1` | `validate` found error-severity problems |
| `2` | Bad arguments, or `init` found an existing file with no TTY to confirm overwrite |
