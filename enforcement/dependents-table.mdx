The Dependents Table audits all of your project's third-party packages in a single panel, checking each one for outdated versions and known security vulnerabilities sourced from the [OSV.dev](https://osv.dev) database. Instead of running separate audit tools for each package ecosystem, you get a unified view covering npm, Go modules, and Python packages side by side.

<Note>
  The Dependents Table requires **Iris Code Pro**.
</Note>

## How to open it

Run **Iris Code: Open Dependents Table** from the command palette.

## Supported package sources

- **npm** - `package.json`
- **Go modules** - `go.mod`
- **Python** - `requirements.txt` and `pyproject.toml`

## Lockfile-aware versions

When a lockfile is present, installed versions are resolved from it rather than from the manifest's version range, so the audit reflects what is actually installed. Supported lockfiles:

- **npm** - `package-lock.json` (v1-v3)
- **pnpm** - `pnpm-lock.yaml`
- **yarn** - `yarn.lock` (classic and Berry)
- **Python** - `poetry.lock` and `Pipfile.lock`

Advisories are filtered to the installed version: a CVE that only affects versions you do not have installed will not be reported.

## Monorepo support

Workspace members declared via npm or yarn workspaces, or in `pnpm-workspace.yaml`, are discovered automatically and their dependencies deduplicated into a single view. Internal `workspace:` / `file:` / `link:` dependencies are excluded from the audit, so only real third-party packages appear.

## What the table shows

For each package, the table displays the installed version, the latest available version, and any known CVEs from the OSV.dev database:

| Package | Installed → Latest | Advisories |
| --- | --- | --- |
| `next` | 14.1.0 → 15.3.2 | — |
| `express` | 4.18.2 → 4.21.2 | CVE-2024-29041 |
| `jsonwebtoken` | 8.5.1 → 9.0.2 | CVE-2022-23529, CVE-2022-23540 |

Each advisory shows a severity badge, the advisory id, a **confidence label**, and the **fixed-in version** where one exists.

## Confidence labels

Every advisory carries a confidence label so you know how actionable it is at a glance:

| Label | Meaning |
| --- | --- |
| `fix-available` | A patched version exists - upgrade to the fixed-in version to resolve it |
| `direct` | The vulnerable package is a direct dependency of your project |
| `transitive` | The vulnerable package is pulled in indirectly by another dependency |
| `manual-review` | The advisory could not be matched precisely to your installed version - review it yourself |

## Unverifiable advisories are never shown

Iris Code never displays advisory data it cannot verify. Any record that lacks a verifiable id or severity is dropped rather than shown, and the panel footer discloses how many were skipped ("N advisories skipped (unverifiable)"). Duplicate records for the same underlying advisory are collapsed into one row.

If the lookup for a dependency fails (network error, registry outage), that dependency is marked **not checked** - it is never silently assumed clean.

## CVE badge in the sidebar

The Dependencies section of the Workspace tab shows a summary pill of the last cached dependency scan: a green "no known CVEs" badge when the scan came back clean, or the number of vulnerable packages plus the highest severity found. The badge reads only from the local cache - it never triggers a network lookup itself. Run a scan from the Dependents Table (or `iris deps` in the terminal) to refresh it.

## Caching

Results are cached locally for 24 hours at `.iris-cache/dependents.json`. Re-opening the panel is instant without a new network round-trip. Iris Code automatically adds `.iris-cache/` to your `.gitignore` on the first write.

## No token required

Vulnerability lookups query OSV.dev directly. There is no token to create, no rate-limit setup, and no in-panel token button - it works out of the box. Only the package name, ecosystem, and version are sent for each lookup - never your code, file paths, or project names.

<Tip>
  If the advisory lookup for a package errors, the table still loads with version data - the affected package is marked "not checked" so you know its advisory status is unknown rather than clean.
</Tip>
